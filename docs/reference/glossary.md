# Glossary

Terms that appear throughout the Verum documentation, with precise
meanings.

## A

**Affine type** ‚Äî A type whose values may be used at most once; the
compiler forbids duplication but allows discarding. Declared with
`type affine T is ...`.

**AOT** ‚Äî Ahead-of-Time compilation. Verum's LLVM backend that produces
native binaries. Tier 3 of the execution model.

**Aliasing** ‚Äî Multiple references to the same value. Verum's rules:
any number of immutable (`&T`), or exactly one mutable (`&mut T`).

**Attribute** ‚Äî A compile-time annotation written with `@` prefix, such
as `@derive(Clone)` or `@verify(smt)`.

## C

**Capability** ‚Äî (1) In the context system, a typed effect like
`Database` or `Logger`. (2) In CBGR, a bit indicating a reference's
allowed operations (`Read`, `Write`, `Admin`).

**CBGR** ‚Äî Capability-Based Generational References. Verum's default
memory safety mechanism: ~15 ns runtime check per deref.

**Cog** ‚Äî A Verum package. Distributable archive containing VBC,
metadata, and optional proof certificates.

**Context** ‚Äî A typed capability requested via `using [...]` and
provided via `provide`. Replaces globals / thread-locals.

**Contract** ‚Äî A function specification: `where requires`, `where
ensures`, loop `invariant`, `decreases`.

**Cubical** ‚Äî Cubical type theory. Verum's foundation for path types
and higher-inductive types with computational univalence.

## D

**Dependent type** ‚Äî A type that depends on a value. ŒÝ-types (function
types whose return depends on an argument), Œ£-types (pairs whose
second depends on the first), path types.

**Derive** ‚Äî Automatic generation of protocol implementations via
`@derive(P)`.

## E

**Epoch** ‚Äî A counter in CBGR that prevents generation wraparound. Each
thread advances its epoch periodically.

**Escape analysis** ‚Äî Compile-time inference of whether a reference can
outlive the scope that produced it. Used to promote `&T` to `&checked T`.

**Execution tier** ‚Äî 0: Interpreter, 1: Baseline JIT, 2: Optimising
JIT, 3: AOT, 4: GPU. See **[Runtime tiers](/docs/architecture/runtime-tiers)**.

## F

**FatRef** ‚Äî 32-byte reference for unsized types (slices, `dyn`).

**FFI** ‚Äî Foreign Function Interface. `extern "C"` blocks; C ABI only.

## G

**Generation** ‚Äî A 32-bit counter in an allocation's CBGR header,
incremented on free. A reference with a stale generation is rejected.

**Generic** ‚Äî A type or function parameterised by types, const values,
lifetimes, kinds, or contexts.

## H

**HIT** ‚Äî Higher-Inductive Type. A sum type with path constructors ‚Äî
i.e., constructors that produce equalities, not just values.

**HoTT** ‚Äî Homotopy Type Theory. The family of type theories in which
path types and univalence are first-class. Verum implements a cubical
fragment.

## I

**Incremental compilation** ‚Äî Re-compiling only the functions whose
source or dependencies changed. Verum's default mode.

**Invariant** ‚Äî A property that holds at every iteration of a loop,
stated with `invariant` clause.

**`is`** ‚Äî Reserved keyword. Used in `type T is ...` and as a pattern /
type-test operator.

## L

**Linear type** ‚Äî A type whose values must be used exactly once.
Stronger than affine.

**LSP** ‚Äî Language Server Protocol. `verum lsp` implements LSP 3.17.

## M

**Mathesis** ‚Äî Verum's ‚àû-topos of formal theories; a research-facing
stdlib module.

**Meta function** ‚Äî `meta fn` ‚Äî a function that runs at compile time.

**MIR** ‚Äî Mid-level IR used during optimisation phases, between AST
and VBC.

**Module** ‚Äî A `.vr` file or a directory with `mod.vr`. Imported with
`mount`.

## N

**Nursery** ‚Äî A structured-concurrency scope. Owns a set of tasks; does
not return until they are all joined or cancelled.

## O

**Obligation** ‚Äî A proof requirement generated by the verifier, such
as "prove `xs[i]` is in bounds" or "prove the postcondition holds."

**Orphan rule** ‚Äî Cross-cog protocol-implementation coherence: an
`implement P for T` is valid only if `P` or `T` is from the defining
cog.

## P

**Path type** ‚Äî `Path<A>(a, b)` ‚Äî a proof that `a` and `b` are equal
in type `A`, treated as data that can be transported.

**Portfolio (verification)** ‚Äî Running Z3 and CVC5 in parallel on the
same obligation and cross-validating.

**Protocol** ‚Äî A trait / interface. Declared as `type P is protocol { ... }`.

## R

**Refinement type** ‚Äî A type plus a predicate that every value must
satisfy: `Int { self > 0 }`.

**Refinement reflection** ‚Äî Extending the refinement vocabulary by
making `@logic` functions available as SMT axioms.

## S

**Semantic honesty** ‚Äî Verum's rule that type names describe meaning,
not implementation: `List` (not `Vec`), `Text` (not `String`).

**SMT** ‚Äî Satisfiability Modulo Theories. The class of solvers Z3 and
CVC5 belong to.

**SSA** ‚Äî Static single assignment form. Used by CBGR analysis.

**Structured concurrency** ‚Äî Concurrency organised into lexical scopes
(nurseries) that own their tasks' lifetimes.

## T

**Task** ‚Äî A unit of async execution. Spawned with `spawn`; joined via
a `JoinHandle`.

**Theorem** ‚Äî A proved statement declared with the `theorem` keyword.

**ThinRef** ‚Äî 16-byte reference for sized types.

**Tier** ‚Äî See "Execution tier" and "Reference tier."

**Transport** ‚Äî Moving a value along a type-level path. Key operation
in cubical type theory.

## U

**Univalence** ‚Äî The axiom "type equivalence is type equality." Verum
implements it computationally in its cubical fragment.

## V

**V-LLSI** ‚Äî Verum Low-Level System Interface. Zero-FFI kernel layer.

**VBC** ‚Äî Verum ByteCode. The compiler's unified intermediate
representation. Every tier runs on VBC.

**VCS** ‚Äî Verum Conformance Suite. The comprehensive language test
suite at `vcs/`.

## Z

**Z3** ‚Äî Microsoft Research's SMT solver. Verum's default; strong at
LIA, bitvectors, arrays.
